# Spec: コンポーネント分割

## 概要

このトラックでは、既存の`App.tsx`に実装されたUIとロジックを、実装済みの機能を壊さないように注意深くコンポーネントに分割します。コード管理のしやすさと可読性の高い構造を目指し、機能ごとのコンポーネント化を主眼に置きつつ、Atomic Design原則に基づいた細粒度での分割を行います。状態管理は最小限のReact State (`useState`, `useReducer`) を活用し、命名規則にはPascalCaseをコンポーネント名に、Kebab-caseをファイル名に採用します。

## ゴール

- `App.tsx`のコードベースを機能ごとに分割された複数のコンポーネントにリファクタリングする。
- 分割後もアプリケーションの既存機能が正常に動作することを保証する。
- コードの可読性とメンテナンス性を向上させる。
- 将来的な機能拡張やUIの変更に柔軟に対応できる基盤を構築する。

## 機能要件

### 1. コンポーネントの機能的分割

- メモアプリの主要な機能（例: メニューバー、エディタ、ステータスバー、ファイル操作関連のUI）をそれぞれ独立したコンポーネントとして分割する。
- UI要素とビジネスロジックを可能な限り分離し、各コンポーネントが単一の責任を持つように設計する。

### 2. コンポーネントの粒度

- Atomic Design原則に基づき、アトム、モレキュール、オーガニズムといった階層でコンポーネントを分割する。
- 例えば、ボタン、入力フィールドなどの基本的なUI要素をアトムとして抽出し、それらを組み合わせてより複雑なコンポーネントを構築する。

### 3. 状態管理

- 各コンポーネントの状態は、必要に応じて`useState`や`useReducer`といった最小限のReact State管理フックで完結させる。
- コンポーネント間のデータ受け渡しは、`props`を介して明示的に行うか、必要に応じてReact Context APIの活用を検討する。

### 4. 命名規則とファイル構造

- コンポーネント名には`PascalCase`（例: `MenuBar`, `EditorArea`）を使用する。
- コンポーネントのファイル名には`Kebab-case`（例: `menu-bar.tsx`, `editor-area.tsx`）を使用する。
- 各コンポーネントは、関連するファイル（スタイル、テストなど）とともに専用のディレクトリに配置する（例: `src/components/MenuBar/index.tsx`）。

## 非機能要件

- **可読性:** 分割されたコンポーネントのコードは、明確な役割と構造を持ち、容易に理解できること。
- **メンテナンス性:** コンポーネントの変更が他の部分に与える影響を最小限に抑えること。
- **パフォーマンス:** コンポーネント分割後も、アプリケーションのレンダリングパフォーマンスが維持されること。

## 受け入れ基準

- `App.tsx`が複数の機能コンポーネントに分割されていること。
- 分割されたコンポーネントが`src/components`ディレクトリ以下に、適切なファイル構造と命名規則で配置されていること。
- `npm run dev`でアプリケーションを起動し、既存のすべてのUIとロジックが正常に動作すること。
- 各コンポーネントが、その責任範囲内で状態を管理し、Propsを通じて適切にデータをやり取りしていること。
- コードレビューにより、コンポーネントの分割が機能分離と可読性向上に寄与していることが確認されること。

## スコープ外

- ReduxやZustandなどの大規模な状態管理ライブラリの導入。
- 既存のUI/ロジックの大規模な機能追加（リファクタリングを目的とするため）。
- 複雑なアニメーションやトランジションの実装。
